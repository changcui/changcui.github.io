于是我的任务变成了显卡采购建议收集，我收集了各个显卡的数据，给出显卡采购的方案，虽然我们组比较菜，只能用别人用剩下的显卡，可总比我本机的 1060 要强一些吧。

昨天遇到的 hang 的问题，今天看了个 pytorch 的 issue，起码有点感性的认识了。这个问题的原因是 pytorch 的 dataloader 当 num_workers 不为0的时候，会调用 multiprocessing 来生成多个进程并行地处理数据，之所以需要多个进程而不是多个进程，是因为 python 的 interpreter 不允许多线程并行，有一个叫 GIL (global interpreter lock) 的机制在，该机制的目的是保护线程安全，比如引用计数等，multiprocessing 产生多个进程默认是调用的 fork 方法，fork 是在当前进程的基础上复制出一个相同的进程，当与一些第三方库，比如 numpy，opencv 混用的时候，第三方库会使用多线程来处理数据，fork 的时候会将多线程的信息也复制一遍，导致死锁。

解决方案一，让 num_workers = 0，不使用多进程，自然会比较慢，但不会死锁。

解决方案二，让 multiprocessing 的默认产生进程的方法是 spawn，和 fork 的差别在于， spawn 是从头产生一个新的进程，比 fork 效率要低。

解决方案三，让 numpy，opencv 等第三方库的调用线程数为1。

解决方案四，随缘更新，指不定哪个版本就解决了这个问题。

关于 GIL，这个链接讲得很好，GIL 是有历史原因的，当初是为了兼容一些线程不安全的 C 拓展库，在易用性和效率上做了折衷，也是 python 这么流行的原因之一。在后来的 python3 中也保留这个特性，因为如果要删除的话，会跑得比 python2 慢，算是一种妥协吧。

这个链接也探讨了 GIL 的问题，并针对 python 上的并行做出了分析，多线程和多进程之间的差别，以及 fork 和 spawn 的差别，并给出了一个新的解决方案。
