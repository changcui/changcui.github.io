Git

我又开始学 git 了，不知道这是第几遍了，git 的命令实在是太多了，今天看了大概三章，可能因为划水没有活干吧，学一些自己觉得一定有用的东西一定没错，这次就没必要从最基础的总结了，总结一下自己觉得比较有用的吧。

  • git 直接记录数据的快照，而不是记录其差异
  • git 有三种状态，已提交，已修改，已暂存
  • git config --global user.name "John Doe"
  • git config --global user.email johndoe@example.com
  • git config --global core.editor emacs
  • 检查配置信息：git config --list
  • 获取帮助：git help <verb>
  • 状态简览：git status --short
  • 比较已修改文件和原文件的差异：git diff
  • 比较暂存文件和原文件的差异：git diff --staged
  • 从 git 中删除，而不删除原文件（取消追踪）：git rm --cached foo
  • git log -p git log -2 git log --pretty=oneline --graph
  • 提交信息写错，文件忘记添加：git commit --amend
  • 取消暂存：git reset HEAD <file>
  • 撤销修改（不可恢复）:git checkout -- <file>
  • 添加远程仓库：git remote add <shortname> <url>
  • 拉取信息（拉群所有信息，但不会创建分支）:git fetch <remote-name>
  • 远程仓库重命名、删除:git remote rename, rm
  • 添加别名：git config --global alias.co checkout
  • 新建分支：git branch <name>
  • 删除分支：git branch -d <name>

git 的原理：整个 repo 里面会生成三种节点，分别是 blob, commiter, tree，tree 由 blob 的指针组成，commiter 有 tree 的指针，上一个 commiter 的指针，还有提交人，创作人等信息，每次 commit 会新创建一个 commiter 并指向上一个 commiter。

Docker

因为公司的网络训练在集群上，用的 kubeflow 管理，每个训练任务的 pipeline 大概要分为如下几步：

  1. 写代码
  2. 写 dockfile 打包环境，build push 到远端
  3. 讲数据集传到指定位置
  4. 使用 kubectl 调用 config 文件执行 build 好的远端 image

以我对 docker 的理解，它应该是个创建 container 的工具，而 container 是一个轻量化的虚拟化方案，对比虚拟机而言。

所以我的任务是学会如何将代码的环境在 dockfile 中配置好，并且 build push

有几个需要注意的点：

  • push 有可能会失败，需要 docker login
  • from 继承 image 在 build 的时候会下载很久，可以替换镜像加速下载
  • build 好的 image 是看不到里面的内容的
  • docker 命令调用都是使用 hash value 而不是 name + tag，类似于 git 的 log
  • 使用 docker 需要 sudo 权限

kubectl

kubectl 看起来像是一个简单的脚本，把 docker 的一些功能又封装了一层，因为其安装只需要下载二进制文件即可。

使用前需要进行配置，最好的方案是直接配置到 .kube/config 这个文件里，可以减少很多命令量。

公司在集群上预安装了 kubeflow，执行一些操作的时候往往要加上-n kubeflow 的后缀。

删除 pod 的时候，使用-f foo.yaml最方便，和 apply 用的是同一个配置文件。

打印 log 的时候，可以加 -f 这个参数来持续打印，但不会每秒都有回执，默认应该是两秒一次。
